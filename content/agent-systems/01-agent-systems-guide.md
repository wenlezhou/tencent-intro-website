# AI智能体系统实战

## 什么是AI智能体？

AI智能体是能够自主规划、使用工具、管理状态并完成复杂多步任务的AI系统。与简单的问答不同，智能体能够：

- **自主决策**：根据目标选择行动方案
- **使用工具**：调用外部API、搜索信息、操作文件
- **管理状态**：跟踪进度，跨步骤保持一致性
- **自我纠正**：检测错误并调整策略

---

## 第一部分：智能体核心能力

### 1. 长周期推理与状态跟踪

现代AI智能体能够处理需要多个步骤、甚至跨越多个会话的复杂任务。关键是**专注于增量进展**——稳步推进几件事，而非尝试同时处理所有事情。

**上下文感知能力**：部分模型能跟踪剩余上下文窗口（"token预算"），在接近限制时自动保存进度。

**管理长周期任务的策略：**

```
你的上下文窗口会在接近限制时自动压缩，使你能从上次停下的地方继续工作。
因此，不要因为token预算担忧而过早停止任务。当接近预算限制时，
在上下文窗口刷新前将当前进度和状态保存到记忆中。
始终保持最大程度的持久性和自主性，完整完成任务。
```

### 2. 多上下文窗口工作流

当任务需要跨越多个上下文窗口时，以下策略能确保连续性：

**策略一：首个窗口搭建框架**
- 第一个上下文窗口用于：编写测试、创建初始化脚本
- 后续窗口迭代待办列表

**策略二：结构化测试跟踪**
- 让AI在开始工作前创建测试
- 用结构化格式（如`tests.json`）跟踪测试状态
- 提醒AI测试的重要性

**策略三：初始化脚本**
- 创建`init.sh`等脚本优雅启动服务器、运行测试套件
- 从新上下文窗口继续时避免重复工作

**策略四：状态管理**

| 数据类型 | 推荐格式 | 原因 |
|----------|----------|------|
| 结构化数据 | JSON | 模式明确，易于解析 |
| 进度笔记 | 自由文本 | 灵活记录上下文 |
| 代码变更 | Git | 日志完整，可恢复 |
| 任务状态 | 结构化格式 | 清晰跟踪完成度 |

### 3. 子智能体编排

AI能识别任务何时需要委派给专门的子智能体，并主动进行编排。

**适合委派子智能体的场景：**
- 任务可以并行执行
- 需要隔离的上下文
- 独立的工作流，不需要共享状态

**不适合委派的场景：**
- 简单任务（直接处理更快）
- 需要跨步骤保持上下文的任务
- 单文件编辑

**控制子智能体使用：**

```
在任务可以并行运行、需要隔离上下文或涉及不需要共享状态的独立工作流时，
使用子智能体。对于简单任务、顺序操作、单文件编辑或需要跨步骤
保持上下文的任务，直接处理而非委派。
```

## 第二部分：工具使用

### 4. 明确指令触发行动

AI有时会给出建议而非直接实施，即使实施才是你的本意。

**主动执行模式：**

```xml
<default_to_action>
默认实施更改而非仅提供建议。如果用户意图不明确，
推断最有用的可能行动并继续，使用工具发现缺失细节而非猜测。
</default_to_action>
```

**保守执行模式：**

```xml
<do_not_act_before_instructions>
除非明确指示进行更改，否则不要直接实施或修改文件。
当用户意图模糊时，默认提供信息、研究和建议。
</do_not_act_before_instructions>
```

### 5. 并行工具调用优化

现代AI模型能够并行运行独立的工具调用：
- 研究时并行运行多个搜索
- 同时读取多个文件
- 并行执行独立命令

```xml
<use_parallel_tool_calls>
如果你打算调用多个工具且工具调用之间没有依赖关系，
请并行执行所有独立的工具调用。优先同时调用工具而非顺序调用，
以提高速度和效率。但如果某些工具调用依赖之前的调用结果，
则必须顺序执行。永远不要使用占位符或猜测缺失的参数。
</use_parallel_tool_calls>
```

### 6. 思维与推理控制

**自适应思维（Adaptive Thinking）**：AI动态决定何时思考及思考多少，根据查询复杂度和`effort`参数自动校准。

| 思维模式 | 适用场景 |
|----------|----------|
| 高effort + 自适应思维 | 复杂多步推理、智能体行为 |
| 中等effort | 常规任务 |
| 低effort | 简单直接的问题 |

**引导思维行为：**

```
收到工具结果后，仔细反思其质量并确定最佳后续步骤再继续。
利用你的思维来规划和迭代，然后采取最佳下一步行动。
```

**减少思维触发：**

```
扩展思维会增加延迟，只在对答案质量有显著提升时使用——
通常是需要多步推理的问题。有疑虑时，直接响应。
```

## 第三部分：安全与质量控制

### 7. 平衡自主性与安全性

AI智能体可能执行难以撤销或影响共享系统的操作。关键是在自主性和安全性之间取得平衡。

**需要确认的操作类型：**

| 类型 | 示例 |
|------|------|
| 破坏性操作 | 删除文件/分支、清空数据库、rm -rf |
| 难以撤销的操作 | git push --force、git reset --hard |
| 对他人可见的操作 | 推送代码、评论PR、发送消息 |

**安全提示词：**

```
考虑操作的可逆性和潜在影响。鼓励执行本地、可逆的操作，
如编辑文件或运行测试。但对于难以撤销、影响共享系统
或可能具有破坏性的操作，请在执行前询问用户。
```

### 8. 避免过度工程化

AI智能体有时倾向于：
- 创建额外文件
- 添加不必要抽象
- 构建未请求的灵活性
- 过度文档化

**最小化过度工程：**

```
避免过度工程化。只做直接请求或明显必要的更改。保持方案简单聚焦：

- 范围：不添加功能、重构代码或做超出要求的"改进"
- 文档：不为你未修改的代码添加文档字符串或注释
- 防御性编码：不为不可能发生的场景添加错误处理
- 抽象：不为一次性操作创建辅助函数或工具
```

### 9. 最小化幻觉

```xml
<investigate_before_answering>
永远不要对你未打开的代码进行推测。如果用户引用了特定文件，
你必须在回答之前阅读该文件。在回答关于代码库的问题之前，
确保调查并阅读相关文件。除非你确定正确答案，
否则不要在调查之前对代码做出任何声明——
提供有根据的、无幻觉的答案。
</investigate_before_answering>
```

### 10. 避免硬编码和应试

AI有时会过度专注于通过测试，而牺牲通用解决方案：

```
请使用标准工具编写高质量的通用解决方案。不要创建辅助脚本
或变通方法来更高效地完成任务。实现一个对所有有效输入
都能正确工作的解决方案，而不仅仅是测试用例。
不要硬编码值或创建仅对特定测试输入有效的解决方案。

测试是为了验证正确性，而非定义解决方案。
提供遵循最佳实践和软件设计原则的有原则的实现。
```

## 第四部分：模型上下文协议（MCP）

### 11. MCP入门

模型上下文协议（Model Context Protocol）是一种开放标准，用于连接AI与外部服务。

**三大核心原语：**

| 原语 | 用途 | 示例 |
|------|------|------|
| **工具（Tools）** | 让AI执行操作 | 搜索数据库、调用API |
| **资源（Resources）** | 为AI提供数据 | 文件内容、数据库记录 |
| **提示（Prompts）** | 预设提示词模板 | 常用任务的标准化提示 |

**用Python构建MCP服务器：**


### 12. MCP进阶

**高级实现模式：**
- **采样（Sampling）**：让MCP服务器请求AI完成子任务
- **通知（Notifications）**：实时推送状态更新
- **文件系统访问**：安全地访问本地/远程文件
- **传输机制**：HTTP、SSE、WebSocket等

## 第五部分：研究能力

### 13. 结构化研究方法

AI智能体能够有效查找和综合多源信息。最佳研究结果的关键：

1. **提供明确的成功标准**：定义什么算作成功答案
2. **鼓励来源验证**：跨多个来源验证信息
3. **使用结构化方法**：

```
以结构化方式搜索信息。收集数据时，发展多个竞争假设。
在进度笔记中跟踪置信水平以提高校准。定期自我批评你的方法和计划。
更新假设树或研究笔记文件以持久化信息并提供透明度。
系统性地分解这个复杂研究任务。
```
